# Bug Audit — 2026-05-02

Scope: `net/crates/net/` (477 Rust files). Five parallel hunters scanned by functional area:
cortex, channel/redex, compute/traversal/contested/continuity, sdk/bindings, and
state/subnet/behavior/identity/netdb/config. Findings consolidated and ranked below.

Severity ladder:
- **Critical** — panic/corruption/security failure, no caller-visible signal
- **High** — incorrect behavior on a realistic path
- **Medium** — edge case or contention-only failure
- **Low** — latent / footgun / by-design / astronomical

## Critical / High

### Lost events on partial replay failure
**File:** `src/adapter/net/compute/migration_target.rs:400-426`
```rust
while let Some(event) = state.pending_events.remove(&next_seq) {
    to_replay.push(event);   // events removed BEFORE delivery
    next_seq += 1;
}
...
for event in &to_replay {
    self.daemon_registry.deliver(...)?;   // early-return on failure
}
if let Some(last) = to_replay.last() { state.replayed_through = last.link.sequence; }
```
**Failure:** If `deliver()` fails on event N of M during `replay_events` / `buffer_event`,
the `?` returns early. All events in `to_replay` were already removed from
`pending_events`, but `replayed_through` is never updated. On retry, those events are
gone. Daemon state is permanently corrupted. **Severity: high.**

### Silent eviction during tail backfill (no `Lagged`)
**File:** `src/adapter/net/redex/file.rs:820–892`
```rust
let start = state.index.partition_point(|e| e.seq < from_seq);
let backfill_count = state.index.len() - start;
```
**Failure:** A subscriber requesting `from_seq=0` after retention has dropped the head
gets only the currently retained tail with no `Lagged` signal — `partition_point`
returns the first retained entry with `seq >= from_seq`, so the "history was lost"
condition is never detected. Documented behavior on `read_range` but `tail`'s doc
explicitly promises `Lagged` on backfill overflow only. Should also fire on
seq-below-retained. **Severity: high (silent data loss for resume-from-snapshot
consumers).**

### Index task exits permanently after `Lagged`
**File:** `src/adapter/net/redex/index.rs:131–139`
```rust
Some(Err(e)) => { tracing::debug!(error = %e, "RedexIndex tail error; continuing"); }
```
**Failure:** `RedexError::Lagged` is logged at `debug` and the loop continues. After
`Lagged` the underlying watcher is dropped from the file, so the next `tail.next()`
yields `None` and the index task exits permanently with the index permanently
incomplete. Keys missing post-`Lagged` stay missing forever. **Severity: high (silent
index corruption).**

### Node binding `u32 → u8` truncation on member index
**File:** `bindings/node/src/groups.rs:664, 672`
```rust

    self.inner.member_role(index as u8)...
```

### `NetBuilder::identity()` silently overrides `entity_keypair`


### Python bindings hold GIL across blocking compute ops
**File:** `bindings/python/src/groups.rs:257-272, 354-366, 464-476`
```rust
fn scale_to(&self, n: u8) -> PyResult<()> { self.inner.scale_to(n).map_err(group_err) }
fn on_node_failure(&self, failed_node_id: u64) -> PyResult<Vec<u8>> { ... }
fn sync_standbys(&self) -> PyResult<u64> { ... }
```

**Failure:** Module docstring promises sync because "PyO3's `Python::attach` is safe
from any tokio worker." But `scale_to`, `on_node_failure`, `promote`, `sync_standbys`

can block on internal locks for ms–s. None wrap with `py.allow_threads(...)`. A Python
program with a watchdog thread or asyncio loop is frozen for the duration. **Severity:
high.**

### Lossy `as_f64` for all numeric ordering in rules
**File:** `src/adapter/net/behavior/rules.rs:118-120`
```rust
let a = a.as_f64()?;
let b = b.as_f64()?;
a.partial_cmp(&b)
```
**Failure:** Rule conditions `Lt`/`Gt`/`Le`/`Ge` against `serde_json` u64/i64 fields
>2^53 (e.g., timestamps in ns, byte counts) silently truncate. Two adjacent values
like `9_007_199_254_740_992` and `9_007_199_254_740_993` compare as `Equal`; a `Gt`
guard on a quota or sequence number returns `false` and the rule fails to fire. NaN /
Infinity (legal under arbitrary_precision) yield `None`, silently masking the bug.
**Severity: high.**

### Snapshot-store retention drops high-water mark on `remove`
**File:** `src/adapter/net/state/snapshot.rs:592-594`
```rust

    self.snapshots.remove(entity_id.as_bytes()).map(|(_, s)| s)
}
```
**Failure:** A stale producer racing a retention-`remove` re-stores an older snapshot
at lower `through_seq` than the just-removed one — concrete ABA / state rollback. The
regression test `cr17_remove_does_not_carry_forward_through_seq_high_water_mark` pins
this as a documented limitation; multi-producer paths can hit it without the locking
the test enforces. **Severity: high.**

### Untrusted-input panics in subnet config
**Files:**
- `src/adapter/net/subnet/assignment.rs:73-77, 124-128`
- `src/adapter/net/subnet/id.rs:43-48`
```rust

assert!(level_value != 0, ...);        // SubnetRule::map
```

### `SdkError::Ingestion(String)` flattens structured `IngestionError`
**File:** `sdk/src/error.rs:85-89`
```rust
impl From<net::error::IngestionError> for SdkError {
    fn from(e: net::error::IngestionError) -> Self {
        SdkError::Ingestion(e.to_string())
    }
}
```
**Failure:** `IngestionError` has structured variants (`Backpressure`, `Sampled`,
`Unrouted`, shutdown). The SDK already exports `SdkError::Backpressure` and
`SdkError::Shutdown` with rich semantics for stream-side errors, but ingestion-side
mappings stringify everything. A user calling `node.emit(...)` on a backpressured bus
gets `SdkError::Ingestion("backpressure: ring buffer full")` and must string-match to
distinguish from `Sampled` (event deliberately dropped, retry pointless) or `Unrouted`
(topology transient). **Severity: high — looks ergonomic but forces wrong remediation.**

## Medium

### Observable seq rollback via `next_seq()`
**File:** `src/adapter/net/redex/file.rs:393, 440, 509, 608, 648, 706`
```rust
let seq = self.inner.next_seq.fetch_add(1, Ordering::AcqRel);
// ... disk write ...
self.inner.next_seq.fetch_sub(1, Ordering::AcqRel); // on disk failure
```
A reader calling `RedexFile::next_seq()` (line 358 — `Ordering::Acquire` outside the
state lock) can observe the temporarily incremented value before the rollback completes
on disk failure. Window where external observers (metrics, snapshot logic, an
`IndexStart::FromSeq(next_seq())` re-tail) believe a seq exists that was never durably
appended.

### Disk I/O held under `parking_lot::Mutex`
**Files:** `src/adapter/net/redex/file.rs:475–565, 678–758, :987`
`append_batch` calls `disk.append_entries_at(...)` and `sweep_retention` calls
`disk.compact_to(...)` while holding `state.lock()` (a non-yielding parking_lot
Mutex). All concurrent appenders, `tail` registrations, `read_range`, `len`,
`is_empty`, and `close` block on the same lock during disk I/O. Latency / starvation,
not correctness.

### `ingest_typed` post-`open` race vs foreign same-origin writers
**Files:**
- `src/adapter/net/cortex/memories/adapter.rs:116-117`
- `src/adapter/net/cortex/tasks/adapter.rs:119-120, 340-341`
```rust
let file = redex.open_file(&name, redex_config)?;
let next_seq = file.next_seq();
```
Concurrent ingests on this channel can append events between `CortexAdapter::open`
(which spawned the fold task with `file.tail(start_seq)`) and `next_seq()` here.
`wait_for_seq(next_seq - 1)` then waits for a moving target — there is a window where
post-read appends with our `origin_hash` (e.g. crashed sibling adapter resumed) can
land *after* `wait_for_seq` returns yet *before* the user's first `ingest_typed`, and
collisions can occur. Doc-comment on `MemoriesAdapter::open` overstates the guarantee.

### DashMap entry guard held across registry I/O
**File:** `src/adapter/net/compute/migration_source.rs:73-101`
```rust
let entry = match self.migrations.entry(daemon_origin) { ... Vacant(entry) => entry };
if !self.daemon_registry.contains(daemon_origin) { return Err(...); }
let snapshot = self.daemon_registry.snapshot(daemon_origin)?...;
entry.insert(...);
```
The DashMap shard write guard from `entry()` is held across `daemon_registry.snapshot()`,
which itself locks an inner `Mutex<DaemonHost>`. Combined with another caller acquiring
the daemon-host mutex first then touching this dashmap shard, deadlock risk. Also,
`take_snapshot` may run user code (`MeshDaemon::snapshot()`) under the held shard
guard, blocking concurrent migrations on co-hashed origins.

### Non-deterministic verdict ordering
**File:** `src/adapter/net/contested/correlation.rs:158-164`
```rust

    .collect::<std::collections::HashSet<_>>().into_iter().collect();
```
`HashSet<u64>` iteration order is randomized per process. `window_failures` is the
`failed_nodes` list passed verbatim into `PartitionRecord::other_side`
(`partition.rs:160`). Two nodes with identical inputs assemble different `other_side`
orderings, causing divergent serialization, divergent reconcile ordering, and replays
that can't be cross-validated.

### `record_execution` window-reset across rule reload
**File:** `src/adapter/net/behavior/rules.rs:1184-1200`
```rust

...
```

`record_execution` is called for rules whose *current* `rate_limit` is `None`

Second audit pass covering areas the original sweep didn't focus on: top-level

### High

#### `ShardManager::activate_shard` double-counts on idempotent calls
**File:** `src/shard/mod.rs:792-800` (with `src/shard/mapper.rs:760-806`)
```rust
pub fn activate_shard(&self, shard_id: u16) -> Result<(), ScalingError> {
    let mapper = ...;
    mapper.activate(shard_id)?;          // returns Ok(()) early when already Active
    self.num_shards
        .fetch_add(1, std::sync::atomic::Ordering::Release);  // unconditional
    Ok(())
}
```
**Failure:** Docstring at line 791 promises "Idempotent: calling on an already-`Active`
shard is `Ok(())`." The mapper honours this — when state is already `Active` it
returns `Ok(())` at `mapper.rs:766` without bumping its own `active_count`. But
`ShardManager` then unconditionally `fetch_add(1)`s `num_shards`. After repeated
activate calls, `num_shards` exceeds the mapper's `active_count` and the actual
shard count, breaking modulo-based shard selection (`select_shard`) and producing
stale routing decisions. **Severity: high.**

#### Silent `event_count` truncation in packet builder
**File:** `src/adapter/net/pool.rs:144, 208`
```rust
let header = NetHeader::new(
    self.session_id, stream_id, sequence, [0u8; NONCE_SIZE],
    self.payload.len() as u16,
    events.len() as u16,            // silent wrap if > 65_535
    flags,
)
```
**Failure:** Protocol-level cap is `MAX_EVENTS_PER_PACKET = 2027`
(`protocol.rs:443`), but the packet builder doesn't enforce it before the cast.
A caller passing >65 535 events wraps to `len % 65536`. The receiver's `validate()`
(`protocol.rs:451`) accepts any `event_count <= 2027`, so a wrapped value like
34 464 fails validation noisily — but a wrapped value below 2027 (e.g. caller passed
67 562 events) passes validation, and the receiver mis-parses the payload because
the actual event_count on the wire mismatches what's encoded. Add
`debug_assert!(events.len() <= MAX_EVENTS_PER_PACKET as usize)` plus a runtime guard
returning an error. **Severity: high (silent corruption).**

### Medium

#### Anti-replay window cleared on large legitimate jumps
**File:** `src/adapter/net/crypto.rs:417, 469-476`
```rust
fn is_valid(&self, received: u64) -> bool {
    if received >= self.rx_counter {
        received.saturating_sub(self.rx_counter) <= Self::MAX_FORWARD  // 65_536
    } ...
}
fn shift_bitmap_up(&mut self, shift: u64) {
    ...
    if shift >= (Self::BITMAP_WORDS as u64) * 64 {  // 1024
        self.bitmap = [0; Self::BITMAP_WORDS];      // wipes prior tracking
        return;
    }
    ...
}
```
**Failure:** `MAX_FORWARD = 65 536` allows accepting future counters far beyond
`WINDOW_SIZE = 1024`. When a legitimate packet jumps the counter forward by more
than 1024, `shift_bitmap_up` zeros the bitmap, leaving the prior ~64 000 counters
unprotected. Any captured packets in that window can be replayed undetected. Either
tie `MAX_FORWARD` to `WINDOW_SIZE`, or document the tradeoff as accepted.
**Severity: medium (replay window of seen packets).**

#### `pushes_since_drain_start` mismatched atomic ordering
**File:** `src/shard/mapper.rs:213-215` (read), `:203-204` (writer)
The reader getter uses `Ordering::Relaxed`; the writer (`set_draining(true)`) uses
`SeqCst` for the reset. On weakly-ordered hardware (ARM), `finalize_draining` can
read a stale counter and falsely conclude the drain has flushed while producers are
still pushing. Reader should be `Acquire` to pair with the release of the reset.
**Severity: medium (architecture-specific).**

#### Silent `shard_id` truncation in JetStream deserialize
**File:** `src/adapter/jetstream.rs:110`
```rust
let shard_id = value.get("s").and_then(|v| v.as_u64()).unwrap_or(0) as u16;
```
A stored `shard_id` of 100 000 silently becomes 34 464 (`100_000 % 65_536`). No
warning. Mis-routes events under the wrong shard at consume time. Either reject the
message or log a deser error; better, store the value as the same width it was
written with.

#### `dedup_state` silently swallows fsync failures
**File:** `src/adapter/dedup_state.rs:221-235`
```rust
let _ = f.sync_all();
```
Disk-full or I/O errors during `sync_all` are discarded. The producer-nonce file
may be partially persisted; the next restart resumes with a stale nonce, defeating
the cross-restart dedup guarantee that's the entire point of the file. Propagate
the error (or at minimum log at warn).

#### `Redis::is_healthy` PING has no enforced timeout
**File:** `src/adapter/redis.rs:465-478`
```rust
async fn is_healthy(&self) -> bool {
    if !self.initialized.load(Ordering::Acquire) { return false; }
    if let Ok(mut conn) = self.get_conn().await {
        redis::cmd("PING").query_async::<String>(&mut conn).await.is_ok()
    } else { false }
}
```
Health entirely depends on `ConnectionManager`'s configured request timeout. If
that's unset or large, a network partition leaves the health check blocked
indefinitely, making the adapter appear "pending" rather than "unhealthy". Wrap in
`tokio::time::timeout(self.config.request_timeout, …)`.

#### `net_num_shards` returns 0 on null handle
**File:** `src/ffi/mod.rs:1069-1078`
Returns `0` for null handle — also a valid (if unusual) shard count. Inconsistent
with sibling FFI functions that return negative `NetError` codes; C callers can't
distinguish null handle from a 0-shard bus. Return `NetError::InvalidHandle.into()`.

### Claims investigated and rejected
Listed so future audits don't re-litigate them.

- **Off-by-one null-terminator in `net_poll`** (`ffi/mod.rs:806/817`) — the bound
  check `response_json.len() + 1 > buffer_len` is correct: when `len + 1 ==
  buffer_len`, the write at `out_buffer.add(len)` lands at the last valid byte.
- **`OnceCell` caches stream-init errors forever** (`adapter/jetstream.rs:131`) —
  the docstring at line 129-130 and `OnceCell::get_or_try_init` semantics confirm
  errors leave the cell uninitialized; subsequent calls retry.
- **`shift_bitmap_up` UB at `bit_shift == 64`** (`adapter/net/crypto.rs:495`) —
  guarded by the `if bit_shift == 0` branch at line 479; the right-shift is always
  in `[1, 63]`.
- **`session_prefix_from_id` XOR collisions** (`adapter/net/crypto.rs:511-515`) —
  irrelevant to security: keys are per-session-direction, so nonce uniqueness only
  needs `(key, nonce)` uniqueness within one key.
- **JetStream batch ordering across concurrent `on_batch`** (`jetstream.rs:262-277`)
  — the per-shard FIFO contract is enforced upstream by the shard manager
  serializing per-shard dispatch; this layer only sees one batch at a time per shard.
